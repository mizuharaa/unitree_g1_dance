# G1 MODEL RECONCILIATION — mjlab (training) vs menagerie (old preview) vs real G1

**Date:** 2026-07-16 · **Author:** Agent E (preview fidelity) · **Status:** feeds Agent A (trust) + Agent D (design)
**Scope:** CPU-only, read-only on the training stack. Every number is extracted from a model file on disk
(`mujoco.MjModel`) or from `pipeline/g1_limits.py` (whose provenance is BeyondMimic `g1.py` + `policy_meta.json`).

## TL;DR

The on-laptop preview used to replay the trained policy on the **menagerie** G1
(`third_party/mujoco_menagerie/unitree_g1/scene.xml`) — a *different* model from the mjlab training model.
Result: the policy looked **washed-out / frozen** (it dances only ~7–27% of the reference amplitude) and the
operator had no faithful preview. This is now fixed. The default preview model is a **faithful mjlab-aligned
model**:

> **`tools/assets/g1_faithful/g1_mjlab_faithful.xml`** = the **official Unitree G1 MJCF**
> (`third_party/unitree_mujoco/.../g1_29dof.xml`, correct hardware inertias, the same source g1_limits treats
> as hardware truth) **+ per-joint ARMATURE patched to the mjlab/BeyondMimic values** (`g1_limits.ARMATURE`)
> **+ XML joint damping/frictionloss zeroed** (mjlab impedance-actuator convention: damping is supplied by the
> deploy PD `kd`, joint friction lives in the actuator model, not the XML) **+ a floor/light** so it stands.

On this model the **same** v7 policy (`data/policies/thriller_v7ank`, obs/PD imported unchanged from
`deploy_runtime`) **dances at ~96–180% amplitude and stays upright**, then falls at **~20.7 s** — right after
the first known hard beat (13–18 s), consistent with the gate's "falls cluster at the hard beats, ankles
saturate" story. On menagerie the same policy froze at 7% and gave the operator no signal.

Built/regenerated by `tools/assets/g1_faithful/build_faithful_model.py`. Wired as the **default** in
`tools/sim_sandbox.py` and `tools/sim_studio.py` (override with `--menagerie` or `--model <xml>`).

## Behavioural evidence (v7 policy, thriller_v7ank, no added latency, tether 0)

| model | policy amplitude (dances) | fall time (full 51.8 s dance) | verdict |
|---|---|---|---|
| **menagerie** (old default) | **7%** (800 steps) / 27% (200 steps) | stood but frozen | washed-out artifact — useless preview |
| **faithful** (new default) | **96%** (800 steps) / 118% (full) | **~20.7 s** (after first hard beat) | dances + falls where the gate says |
| faithful + 60 ms latency | 284% (oscillating) | 2.7 s | latency destabilises hard (flags a real sim2real risk) |

The static "hold the default pose with pure joint-PD, no policy" test is **not** a fidelity criterion — a
fixed-joint humanoid does not self-balance. Menagerie happens to hold it (it is lighter and easier to balance,
the *wrong* direction for fidelity); the faithful model tips forward in ~1.2 s with no policy, exactly as a
real un-actuated humanoid would. Fidelity is judged by the **policy** rollout, above.

## The biggest mjlab-vs-menagerie differences (why the preview was broken)

1. **Per-joint armature (THE fix).** mjlab uses per-joint rotor inertia (0.0036–0.0251); menagerie uses a
   flat 0.01. Armature is reflected rotor inertia — it sets how the policy's PD torques accelerate each joint,
   and mjlab's `kp/kd` are *derived from* these armatures (`kp = armature·(2π·10)²`). A policy trained against
   per-joint armatures, replayed on flat-0.01 menagerie, gets the joint dynamics wrong and washes out. This
   single change is the dominant cause of the "offset / washed-out / frozen" preview.
2. **Total mass (5%).** menagerie is 33.34 kg vs the hardware-correct 35.11 kg — a lighter robot with a
   different CoM, so balance and required torques differ from what the policy trained on.
3. **Joint damping/frictionloss.** menagerie carries 0.3 N·m frictionloss (and official 0.05 damping); mjlab
   puts damping in the PD and friction in the actuator, so the faithful XML zeroes both to avoid
   double-counting the deploy `kd`.
4. **Foot friction.** menagerie 0.6 vs official/faithful 1.0 (see caveat below).

## Residual gaps & honest caveats (still NOT the real robot)

- **Foot friction is NOT pinned to a known mjlab value.** The faithful model inherits the official Unitree
  **1.0**; menagerie uses 0.6; the true mjlab value is unknown without the mjlab model (not installable here —
  see below) and mjlab **domain-randomises** friction during training anyway. Armature (the dominant
  joint-dynamics term) is matched exactly; foot friction is the main un-pinned parameter. Flag for Agent D.
- **Flat torque clamp, not the real T–N curve.** ALL of these sims (incl. mjlab) use a flat effort clamp
  (ankle 50 N·m at any speed). The real G1 (Isaac `UnitreeActuator`) has torque **falling with joint speed**
  plus dry friction. At the two failure beats the ankles are fast **and** high-torque, so every sim here is
  **optimistic** exactly where the robot falls. `pipeline/g1_limits.effective_torque_limit()` models a
  conservative derate for feasibility; the *preview physics* does not — so the preview may still be slightly
  optimistic on ankle authority. (Agent 0 report §5.)
- **`mjlab` is not installed on this laptop** (`ModuleNotFoundError: mjlab`), and we must NOT pull
  `unitree_rl_mjlab` (it pins mjlab 1.2.0 / mujoco-warp 3.5.0 and would break our 1.5.0 lock — Agent 0 §6).
  So the faithful model was **assembled** from the official Unitree MJCF + the mjlab armatures already
  extracted into `g1_limits.py`, not exported from a live mjlab install. It matches the training model on
  inertia + armature + gains; it is not a byte-for-byte copy of mjlab's MJCF (contact/friction fine-tuning
  and domain-randomisation ranges are not reproduced).
- **Latency destabilises hard** (fall at 2.7 s with 60 ms). Worth an Agent A/D look — it may indicate the
  policy is brittle to the measured 40–80 ms sensorimotor delay, or that the faithful contact is stiff.
- This is the **training** model, not the **real robot**. It answers "does the preview match what was
  trained?" (now: yes) — it does **not** by itself answer "will it work on hardware" (that is the calibration
  Agent A owes against the one real datapoint).
